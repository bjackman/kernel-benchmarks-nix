# Secretmem vs. Physical Memory Fragmentation Benchmark

This directory implements the `secretmem-vs-frag` benchmark, a specialized test
suite designed to study the performance characteristics, allocation boundaries,
and system-level impact of **Linux Secret Memory (`memfd_secret`)** under
hostile, fragmented physical memory conditions inside Address Space Isolation
(ASI) target environments.

---

## 1. Core Concepts

### Linux Secret Memory (`memfd_secret`)

`memfd_secret` is a Linux system call (introduced in 5.14) that creates a
file-descriptor-backed memory region. The physical pages backing this region are
**completely removed from the kernel's direct map**. They are only mapped into
the user-space process virtual address space.

- **Security Objective**: Protect highly sensitive data (keys, secrets) against
  speculative execution hardware attacks (like Meltdown, MDS) and standard
  kernel-space disclosure bugs, as even the kernel itself will crash with a page
  fault if it attempts to read these physical addresses.
- **Anti-Fragmentation Impact**: Because the kernel must remove secret pages
  from its direct map, it has to **break down large contiguous page tables**
  mapping the direct map (typically 2MB huge pages on x86*64) into base 4KB
  pages. This causes \_direct map fragmentation*, raising TLB pressure.

### Physical Memory Fragmentation & Compaction

Physical memory managed by the buddy allocator gets fragmented over time into
small, isolated blocks. When a high-order allocation (e.g., contiguous 2MB huge
pages) is requested, the kernel must run **memory compaction** to migrate
movable pages in-place and coalesce free memory into larger contiguous holes.

This benchmark creates a **memory antagonist** inside the guest VM that
intentionally pollutes main memory zones with **pinned, unmovable physical
barriers**, rendering compaction useless and creating a hostile "Swiss Cheese"
physical allocation landscape.

## 2. Methodology & Execution Modes

The benchmark operates in a coordinated multi-process architecture consisting of
a host-spawned **Runner** parent process and an allocation **Worker** child
process.

### Mode A: Baseline Allocation Run (Idle RAM)

This measures standard `secretmem` allocation limits when main memory is clean:

1.  The **Runner** spawns a **Worker** process.
2.  The **Worker** runs a tight loop allocating, pinning, and populating
    `secretmem` chunks of the specified size (default `128 MiB`) using the
    dynamic `libc::SYS_memfd_secret` syscall.
3.  The Worker writes volatile data to every 4KB page to force actual physical
    allocation backing.
4.  The Worker streams its current allocation bytes to the runner via stdout.
5.  Eventually, the guest kernel runs out of memory (OOM) and issues a `SIGKILL`
    (signal 9) to the Worker.
6.  The Runner catches the OOM death, extracts the _last successfully allocated
    bytes_, and logs the run.

### Mode B: Antagonized Allocation Run (Memory Antagonist Enabled)

Enabled using the `--antagonize` flag, this tests `secretmem` behavior under
extreme, artificial physical fragmentation:

```
Physical Memory Layout after fragmentation (Swiss Cheese pattern):
┌───────────┬───────────┬───────────┬───────────┬───────────┐
│ 64KB Pin  │ 64KB Free │ 64KB Pin  │ 64KB Free │ 64KB Pin  │ ... (to 90% of free RAM)
│ (Locked)  │  (Hole)   │ (Locked)  │  (Hole)   │ (Locked)  │
└───────────┴───────────┴───────────┴───────────┴───────────┘
```

1.  **Sysctl Block**: We disable compaction of pinned pages in the system
    configuration (`boot.kernel.sysctl."vm.compact_unevictable_allowed" = 0`).
    This ensures that `mlock`ed pages are treated as absolute, physically
    unmovable barriers.
2.  **Movable Pageblock targeted Pollution**: The Runner allocates 90% of all
    available free RAM in fine-grained **64KB chunks** via standard virtual
    anonymous maps. Since the mapping is anonymous and standard, the kernel
    satisfies page faults from its **Movable** pageblocks. After physical
    backing (page-fault populating), the Runner calls `mlock()` on these pages
    **in-place**. This converts the pages into unmovable (unevictable) memory
    _inside the Movable pageblocks_, successfully defeating the kernel's
    pageblock anti-fragmentation grouping.
3.  **Hole Creation (Swiss Cheese)**: The Runner unmaps every second 64KB chunk
    (`munmap`). This releases 50% of the allocated blocks back to the buddy
    allocator, creating thousands of **64KB free holes** physically separated by
    **64KB unmovable locks**.
    - **The Math**: To form a 2MB huge page, the kernel needs 32 contiguous free
      64KB slots. With our unmovable 64KB barriers scattered randomly at 50%
      density, the mathematical probability of finding a contiguous 2MB block is
      $(0.5)^{32} \times 0.5 \approx 1.3 \times 10^{-6}$ (virtually **absolute
      zero**).
4.  **Buddyinfo Verification**: The Runner parses `/proc/buddyinfo` to collect
    the baseline count of free physical blocks $\ge 2\text{MB}$ in active zones
    (`DMA32`, `Normal`). It verifies the post-fragmentation state to ensure
    either an **80% relative reduction** in huge pages has been achieved, or the
    absolute remaining huge pages is **under 20 blocks** (under 80MB).
5.  **Antagonized Allocation**: Once fragmentation is mathematically verified,
    the Runner launches the Worker OOM cycle. The Worker allocates `secretmem`
    chunks until it gets OOM-killed.
6.  **Self-Cleanup (RAII)**: Upon termination, the Runner drops the fragmenter
    handles, which automatically triggers `munlock` and `munmap` on the
    remaining locked blocks, cleanly releasing all RAM back to the guest system.

## 3. Output Metrics & Facts

The benchmark writes two types of JSON files to the output directory (`OUT_DIR`)
for collection and parsing into the central **Falba** database:

### A. Dynamic Iteration Metrics

Written for _every_ individual test run iteration as
**`secretmem_vs_frag_run_<N>.json`** (e.g. `secretmem_vs_frag_run_1.json`):

- **`allocated_bytes`** (Integer, Unit: `B`): The exact raw physical bytes of
  `secretmem` memory successfully allocated, pinned, and backing-populated
  before OOM termination.
  - _Significance_: Demonstrates system allocation limits. Under `--antagonize`,
    the allocation capacity should drop by approximately the amount of memory
    held locked by the fragmenter (e.g. 768MB under baseline vs 384MB under 90%
    antagonization).

### B. Global Run Summary (Facts)

Written _once_ per successful VM job as **`secretmem_vs_frag_summary.json`**:

- **`antagonized`** (Boolean): Tracks whether the memory antagonist was active
  (`true`) or inactive (`false`) for this experiment run.
  - _Significance_: In Falba, this is stored as a **Fact** (experimental control
    factor), allowing you to group, query, and plot the metric
    `secretmem_allocated_bytes` against `antagonized` states to study
    performance delta.
- **`iterations`** (Integer): The number of consecutive experimental test cycles
  successfully completed in this run.
- **`chunk_size_mib`** (Integer): The size of individual allocation chunks
  (default `128`).

## 4. How to Execute & Test Locally

Since Nix flakes require all build files to be tracked in Git, ensure any
changes are added first.

### A. Run Automated Integration Test

To run a fully automated test (boots guest VM, runs the benchmark in VM sandbox,
writes metrics, shuts down, and runs Falba parsers) from host:

```sh
nix run .#packages.x86_64-linux.impure-tests.secretmem-vs-frag
```

### B. Run Interactive/Persistent VM (Tight Dev Cycles)

To avoid the boot-up and shutdown overhead on every code change, use the
persistent VM approach:

1.  **Boot Guest VM in Interactive Mode** (Host Terminal A):

    ```sh
    nix run .#benchmarks.x86_64-linux.secretmem-vs-frag.in-vm -- -i
    ```

    _(This builds and boots the VM with our system guest configuration, login as
    `root`, and drops you to a console)._

2.  **Execute manually via SSH / Vsock** (Host Terminal B): You can compile and
    run the benchmark directly inside the running guest via shared store paths:
    - **Standard 5-iteration run**:
      ```sh
      ssh -o StrictHostKeyChecking=no root@vsock/3 $(nix build --no-link --print-out-paths .#benchmarks.x86_64-linux.secretmem-vs-frag)/bin/secretmem-vs-frag
      ```
    - **Antagonized single-iteration test with metrics mapped to host**:
      ```sh
      ssh -o StrictHostKeyChecking=no root@vsock/3 "OUT_DIR=/mnt/kbn-output \$(nix build --no-link --print-out-paths .#benchmarks.x86_64-linux.secretmem-vs-frag)/bin/secretmem-vs-frag -- --iterations 1 --antagonize"
      ```
      _(Any JSON metrics generated inside `/mnt/kbn-output` will be visible on
      the host under the temporary directory logged during the interactive VM
      boot)._
